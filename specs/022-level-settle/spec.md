# 022 关卡自然落位与稳定开局

## 背景

旧关卡的积木和目标坐标按动态 Matter 物体编排，进入关卡后依赖短暂重力落到地面或塔顶。上一轮为避免自动坍塌，把这些物体在创建时直接设为静态，导致第 7 关等旧关卡的塔和糖果罐固定在原始悬空位置。

本需求在不重做 25 关数据的前提下，恢复自然落位，同时保留“不开火不坍塌、空射不通关”的稳定性。

## 目标

- 进入关卡后每 80ms 采样积木、糖果罐、宝箱和炸弹的运动状态；累计 650ms 有效采样且连续三次低速后冻结，最迟累计 800ms 采样后兜底。
- 落位期间不生成可操作糖果球，也不执行掉落消除。
- 落位结束后冻结所有可破坏结构，再生成第一颗糖果球并允许玩家操作。
- 糖果球命中或爆炸时，仍只释放命中点附近的局部结构，保留物理连锁。
- 只有被命中区域释放过的普通糖果罐，掉出场地后才算消除。
- 糖果宝箱必须按耐久度命中三次打开，不能通过掉落绕过耐久度。

## 非目标

- 不新增关卡、糖果球、声音、音乐、账号、广告或商业化系统。
- 不重写 1-25 关的坐标和得分门槛。
- 不改变传送门、弹力垫、分裂球、糖果狂热和存档格式。

## 状态与流程

关卡状态按以下顺序流转：

1. `settling`：创建动态积木、目标和炸弹，反馈区显示“糖果塔准备中”。
2. 累计至少 650ms 有效物理采样且连续三次检测为低速后，将当前仍存在的积木、目标和炸弹设为静态；持续运动时最迟累计 800ms 采样后冻结，避免高塔在准备阶段完全坍塌。
3. `ready`：生成第一颗糖果球，恢复现有拖拽、发射和技能流程。
4. `in-flight`：命中目标或积木时，释放半径 250 像素内的结构。
5. `shot-end`：沿用现有回合结束、胜负判断和下一颗球生成流程。

场景销毁或切换后，落位回调不得继续生成糖果球。

## 目标规则

- `TargetState` 增加 `released` 状态，初始为 `false`。
- `releaseImpactArea` 释放目标时同步将 `released` 设为 `true`。
- 普通糖果罐仅在 `released === true` 且位置低于掉落线时触发掉落消除。
- 宝箱无论是否掉出场地，都只通过 `resolveTargetDamage` 的耐久度规则打开。
- 直接命中普通糖果罐仍沿用一次命中消除，不受掉落判断影响。

## 兼容性

- 存档仍使用 `version: 1`，不新增持久化字段。
- 650-800ms 指完成的物理采样预算；低帧率设备的墙钟等待可能更长，但不会因首帧延迟而过早冻结。
- 准备阶段只发生在每次进入或重开关卡时，不插入每次发射之间。
- 第一颗糖果球在落位结束后才生成，因此不会被准备阶段的结构碰撞影响。

## 验收与测试

- 自动化测试覆盖准备阶段不生成糖果球、连续稳定检测与超时兜底、落位结束后冻结结构并生成糖果球。
- 自动化测试直接覆盖 Matter 球体动态创建、冻结和再次释放时恢复有限质量。
- 自动化测试覆盖未释放糖果罐不会因掉落消除，释放后的普通糖果罐可以掉落消除。
- 自动化测试覆盖宝箱不能因掉落绕过三次命中。
- 浏览器实测第 7 关：塔和目标不悬空，静置不会通关，空射不会导致结构坍塌。
- 浏览器实测命中塔身后：局部结构正常倒塌，命中反馈、得分和下一回合正常。
- 执行 `npm test`、`npm run build` 和 `git diff --check`。
